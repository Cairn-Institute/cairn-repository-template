## Repository purpose

This repository is the standard starter template for new CAIRN Institute repositories. When used as the basis for a real project, update this file to describe that project's actual purpose, architecture, and rules before connecting Codex or another AI coding assistant.

## Technical architecture

Describe the real architecture of the project created from this template here: languages, frameworks, services, and how the src, docs, and tests directories are organized.

## Development commands

List the exact commands used to install dependencies, run the project locally, and build it, once this template has been turned into a real project.

## Testing commands

List the exact commands used to run unit tests, integration tests, and any linting or type-checking tools.

## Formatting rules

State the formatter and linter used, and confirm that formatting and linting must pass before a pull request is opened.

## Security requirements

Never commit secrets, API keys, credentials, or production data. Use environment variables and .env.example for configuration guidance. Follow SECURITY.md for vulnerability handling.

## Privacy restrictions

Never use real student, employee, donor, partner, veteran, or health data as test data. Use synthetic or anonymized data only.

## Accessibility standards

User-facing changes should aim for WCAG 2.2 AA. Note any accessibility impact in pull requests.

## Files that should not be changed

Codex should not modify SECURITY.md, CODEOWNERS, .github/workflows files, or production deployment configuration without explicit human instruction and review.

## Documentation requirements

Update README.md and relevant files in docs/ in the same pull request as related code changes.

## Completion-report requirements

When Codex completes a task, it should report what changed, what tests were run and their results, any risks identified, and any follow-up work needed. Codex-generated pull requests must still pass automated checks and human review before merging, and must not be the only reviewer of its own work.
